## 设计意图

门票确认订单页承接门票预定页生成的本地草稿，完成出游日期、联系人、出游人/持卡人信息、可选套餐加购、支付确认和订单详情跳转闭环，是票务样板链路的第二个验收页。优惠券能力仅预留，当前无券业务时不展示入口。

## 页面结构

- 页面根容器：`_pg`
- 页面容器：`PageShell`
- 页面运行时：`usePageRuntime`
- 页面状态订阅：`observer`
- 订单票品卡：展示草稿中的票品、数量和已锁定游玩日期；如需改日期，必须回到门票预定页重新选择票种。
- 加购卡：仅当预定页用户主动选择套餐后展示餐饮套餐加购项；纯门票订单默认不展示，不默认加购。
- 联系人信息卡：联系人姓名、手机号，用于出票和订单通知。
- 入园核验提示：不单独成卡，嵌入门票信息和出游人信息上下文；门票卡提示订单码/购票证件核验，出游人卡提示实名和资格现场核验。
- 出游人信息卡：根据所选票品展示需要填写的游客或持卡人，以“游客 1 / 游客 2 / 持卡人 1”等横向 tab 切换填写，并展示完成进度和对应资格提示。
- 优惠信息卡：有券业务返回时展示折扣和可选优惠券；当前无券业务时不展示。
- 固定支付栏：展示实付金额、优惠金额和支付按钮。
- 页面弹层：确认支付、资格确认和重选日期确认使用微信 modal；有券业务返回时优惠券弹层挂载到 `PageShare`。

## 动态与静态边界

- 草稿读取：页面通过路由 `draftId` 调用 `fetchCheckoutData(draftId)`，优先读取本地草稿；草稿缺失时展示业务空态并引导重新选择门票。
- 日期：确认订单页只展示草稿日期，不在本页直接改日期；点击日期时提示回到上一页重新选择，避免只改日期但票种、价格、库存和入园人信息未刷新。
- 可选加购数量、联系人和出游人：改动后写回草稿，保证刷新或返回后仍可恢复。
- 可选加购：普通提交订单默认 `addonQuantity=0`，确认页不展示套餐；只有从套餐入口确认提交时才携带 1 份套餐进入确认页。
- 票种规则：成人票生成成人实名人；儿童票生成儿童实名人；优惠票生成优待实名人；亲子票生成成人 + 儿童；成人/儿童年卡和家庭年卡按套餐结构生成持卡人。
- 多人填写：tab 数量完全由 `travelers` 数据决定，不写死人数；每个 tab 保留独立姓名、身份证和年卡手机号状态，切换不丢数据。
- 优惠券门槛：能力预留；有券返回时按票品金额 + 加购金额计算，不满足门槛的券在弹层中置灰禁选。
- 支付提交：`submitTicketOrderDraft()` 模拟支付成功并写入本地订单中心。
- 真实接口替换：未来只替换草稿和提交 service，页面表单、校验和跳转协议保持稳定。

## 状态要求

- loading：页面运行时统一承接。
- 草稿缺失：展示统一空态，点击“去选择门票”回到门票预定页，不展示支付栏。
- 表单错误：联系人手机号、出游人姓名、18 位身份证、年卡持卡人手机号逐项校验，重复证件号阻断提交；儿童/优惠资格不确定时用微信 modal 二次确认。
- 支付确认：使用 `showWechatConfirm()` / `showAppModal()` 二次确认。
- 支付成功：写入本地订单并跳订单详情。
- 弹层态：重选日期、资格确认和支付确认使用微信 modal；优惠券仅在接口返回可用券时展示。

## 接口与 Service

| 模块 | service | 失败策略 | 是否阻断页面 |
|---|---|---|---|
| 页面数据 | `fetchCheckoutData(draftId)` | 无草稿时返回草稿缺失态 | 是 |
| 草稿更新 | `updateTicketOrderDraft()` | 本地缓存更新日期、可选加购数量、联系人、出游人；有券时更新优惠券 | 否 |
| 支付提交 | `submitTicketOrderDraft()` | 无草稿时返回空并提示 | 是 |
| 订单中心 | `saveLocalOrder()` | 本地缓存写入 | 是 |

## 交互与跳转

| 元素 | 交互结果 |
|---|---|
| 游玩日期 | 点击后提示回到上一页重新选择；不在确认页直接改日期 |
| 套餐数量 | 仅用户主动选择套餐后展示，使用项目封装步进器调整数量并联动支付金额，可减至 0 |
| 联系人姓名 | 更新联系人姓名 |
| 联系人手机 | 限制 11 位手机号并在提交时校验 |
| 出游人/持卡人 tab | 按票种生成实名信息入口，点击切换当前填写对象 |
| 当前出游人/持卡人表单 | 姓名、证件、年卡手机写回草稿；“同联系人”同步联系人姓名和手机号 |
| 优惠券 | 当前无券业务时不展示；有券返回时打开优惠券弹层 |
| 去支付 | 微信确认弹窗，确认后生成本地订单并跳订单详情 |

## 交互矩阵

| 点位 | 正常结果 | 异常/降级 |
|---|---|---|
| 日期 | `showWechatConfirm()` 提示回到上一页重新选择 | 取消后继续留在确认订单页 |
| 套餐数量 | 用户主动选择套餐后才出现，金额即时重算并写回草稿 | 纯门票订单隐藏加购区 |
| 优惠券 | 有券返回时选择券并重算优惠，可清空不使用 | 当前无券业务时隐藏入口 |
| 出游人 tab | 动态展示游客/持卡人数量、角色和已填/待填状态 | 单人票也按数据渲染一项，不写死人数 |
| 表单 | 联系人和当前出游人本地状态维护并写回草稿 | 提交时逐项提示，重复证件阻断 |
| 资格确认 | 儿童票、优惠票、儿童年卡按证件年龄辅助判断 | 不确定或不满足时弹出资格确认 |
| 支付 | `showWechatConfirm()` 确认后提交 | 取消确认留在当前页 |
| 订单写入 | 本地订单中心新增记录 | 草稿缺失提示重新选择门票 |

## 状态矩阵

| 状态 | 页面表现 |
|---|---|
| 首屏 loading | `usePageRuntime` 默认 loading |
| 正常态 | 展示草稿票品、联系人、实名出游人和支付栏；无券时不展示优惠券入口 |
| 纯门票态 | 不展示套餐加购卡和套餐金额行 |
| 重选日期态 | 微信 modal 提醒日期影响可订门票、价格、库存和入园人信息，确认后优先回退上一页，直开确认页时兜底打开门票预定页 |
| 多人切换态 | 出游人信息以横向 tab 切换，当前仅展开一个表单，顶部显示完成进度 |
| 表单错误态 | toast 提示具体错误 |
| 提交确认态 | 微信 modal 二次确认 |
| 提交成功态 | toast 成功并跳订单详情 |
| 草稿缺失态 | 统一空态提示先选择门票，主动作回门票预定页 |

## 变更记录

### v1.6

- 去除确认订单页可见实现层文案：将“已按票种生成 X 位实名信息”改为“请补充 X 位出游人信息”，避免把系统生成逻辑暴露给游客。
- 日期重选和优惠资格确认弹窗改为游客可理解话术，不再使用“实名信息规则”“园区现场规则”等偏内部表达。

### v1.5

- 修正结算页日期逻辑：确认订单页不再直接打开日期弹层改日期，避免只改日期但票种、价格、库存和入园人信息未刷新。
- 点击游玩日期改为微信确认弹窗，明确日期会影响可订门票、价格、库存和入园人信息，确认后优先回退上一页，直开确认页时兜底打开门票预定页。

### v1.4

- 修正默认加购逻辑：普通门票草稿默认 `addonQuantity=0`，确认订单页不展示套餐加购卡，也不展示 `¥0` 套餐金额行。
- 只有用户在预定页点击套餐入口并确认提交时，草稿才携带 1 份套餐进入确认页；用户仍可在确认页把套餐数量减为 0。

### v1.3

- 出游人信息改为平台式横向 tab 切换：人数由票种规则动态生成，成人票、儿童票、亲子票、家庭年卡等组合自动展开对应游客/持卡人入口。
- 入园核验说明不再独立成卡，改为嵌入门票信息和出游人信息区，避免结算页面割裂。
- 当前出游人表单新增“同联系人”快捷同步联系人姓名和手机号，降低多人填写成本，同时保留证件号独立校验。

### v1.2

- 结算逻辑升级为票品结构驱动：成人票、儿童票、优惠票、亲子票、成人/儿童/家庭年卡分别展示需要填写的出游人或持卡人。
- 联系人与出游人分离，联系人只负责出票通知；出游人按票种校验姓名、身份证、年卡手机号、重复证件和儿童/优惠资格确认。
- 本地订单详情写入实名游客摘要，后续接真实接口时可替换为平台票务实名 DTO。

### v1.1

- 强化门票确认订单页交易闭环：草稿缺失态、动态金额明细、加购数量持久化、联系人持久化、18 位身份证校验。
- 优惠券能力降为低优先级预留：当前无券业务时隐藏入口，订单详情不写入 `0` 优惠金额；有券返回时再启用门槛禁选和“不使用优惠券”清空入口。
- 页面触达主体左右留白收口为 30px，符合页面级默认留白约束。

### v1.0

- 升级为 `commercial-ready` 样板页。
- 确认订单页改为读取预定页 `draftId`，支持本地草稿恢复。
- 日期弹层统一迁入 `PageShare`，避免被固定支付栏压住；优惠券弹层作为预留能力，后续有券业务时再启用。
- 补齐改日期、表单校验、支付确认、订单写入和订单详情跳转。
